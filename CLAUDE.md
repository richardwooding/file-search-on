# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`file-search-on` is a Go CLI that recursively searches a directory and matches files against a [CEL](https://github.com/google/cel-spec) expression evaluated over file metadata and content-type-specific attributes (e.g. `is_pdf && page_count > 10`, `is_markdown && word_count > 500`).

**Markdown front-matter search is a primary feature**, not a side concern — see the dedicated section below before changing anything in `internal/content/markdown.go` or `internal/content/frontmatter.go`.

Module: `github.com/richardwooding/file-search-on`. Toolchain: Go 1.26.2.

## Commands

```sh
go build ./...                                  # build all packages
go build -o file-search-on ./cmd/file-search-on # build the CLI binary
go test ./...                                   # run all tests
go test -race -coverprofile=coverage.out ./...  # what CI runs
go test ./internal/celexpr -run TestEvaluator   # run a single test (regex match)
go vet ./...
go fix -diff ./...                              # preview Go 1.26 modernizers (see below)
go fix ./...                                    # apply them
golangci-lint run                               # CI uses `latest` version
```

### `go fix` (Go 1.26+ modernizers)

Go 1.26 reintroduced `go fix` as a code-modernization tool — it rewrites code to use newer language and stdlib features (e.g. `slices.Contains`, `any`, `min`/`max`, `range` over an integer, `sync.WaitGroup.Go`). See [the announcement](https://go.dev/blog/gofix) for the full set of fixers.

CI runs `go fix ./... && git diff --exit-code` after the build step, so any unapplied modernizer will fail the pipeline. Workflow:

1. Make sure the working tree is clean (`go fix` edits should be a separate commit from feature work).
2. Run `go fix -diff ./...` to preview, or `go fix ./...` to apply. Re-run until idempotent — applying one fix can unlock another.
3. `go test ./...` afterwards; semantic conflicts may need a manual touch-up.

Use `go tool fix help` to list fixers and `go tool fix help <name>` for details on a specific one.

Run the CLI:

```sh
./file-search-on 'is_markdown && word_count > 100' -d ./docs
./file-search-on --list   # list supported attributes & registered content types
./file-search-on mcp      # serve MCP over stdio (see "MCP server" below)
```

If `Expr` is empty it defaults to `"true"` (matches all files). Worker count defaults to `runtime.NumCPU()` when `-w` is 0 or unset.

### Fuzz testing

The high-risk parsers — frontmatter (YAML/TOML/JSON), MP3 ID3v2 + Xing/Info, MKV EBML, MP4 box walker, office/EPUB body extractors (XML + permissive HTML), email body extractor (MIME + multipart + quoted-printable + base64 + mbox separators), PDF body extractor (content streams + font / ToUnicode CMap indirection — `ledongthuc/pdf` is documented as incomplete and panics on adversarial input, so `pdfBody` is wrapped in a defer/recover), CEL expression compile, and the index's gob decoder — have native Go [fuzz targets](https://go.dev/doc/security/fuzz) (`FuzzXxx` functions in `*_fuzz_test.go`).

Workflows:

- **Regular CI** (`ci.yml`): `go test ./...` automatically executes each fuzz target's **seed corpus** (the deterministic inputs added via `f.Add(...)`). Anything that panics during regular CI implies a seed regression — fix or remove the seed.
- **Scheduled fuzz** (`fuzz.yml`): runs each target for 5 minutes nightly at 03:30 UTC. Crashes land in `testdata/fuzz/<FuzzName>/<hash>` inside the failed-run's artifacts; commit those entries to lock in regression coverage. Manual `workflow_dispatch` trigger accepts a `fuzztime` input (e.g. `30m`) for bug-hunt sessions.

Run locally:

```sh
go test -run=FuzzSplitFrontmatter ./internal/content/                          # seed corpus only (fast)
go test -fuzz=FuzzSplitFrontmatter -fuzztime=30s ./internal/content/           # mutate for 30s
go test -fuzz=FuzzSplitFrontmatter -fuzztime=5m ./internal/content/            # bug-hunt session
```

Adding a new fuzz target:

1. Pick a parser that takes adversarial input — bytes from disk, strings from CLI/MCP. Hand-rolled binary parsers are the highest-value targets (the stdlib `debug/elf`, `debug/macho`, `archive/zip` paths are already fuzzed upstream).
2. Add `FuzzXxx(f *testing.F)` to an internal `*_fuzz_test.go` (`package <pkg>`, not `_test`) so unexported functions are reachable.
3. Seed via `f.Add(...)` with a mix of valid inputs and pathological edge cases (empty buffers, truncated headers, gigantic length prefixes).
4. The fuzz body must "never panic" — assertions about returned values are optional; the hard contract is no crash.
5. Add a matrix entry to `.github/workflows/fuzz.yml` so the scheduled run exercises the new target.

## Architecture

Five internal packages compose the pipeline. The first three are tightly coupled by the `FileAttributes` shape; the fifth (`internal/index`) is an optional cache plumbed through `BuildAttributesWith` and `search.Options.Index`:

- **`internal/content`** — pluggable content-type detection. Each type (markdown, json, xml, html, pdf, image variants including HEIC, text, csv, epub, office variants, audio variants, video variants, archive variants, binary variants, email variants, source-code variants for 18 languages) implements the `ContentType` interface (`Name`, `Extensions`, `MagicBytes`, `Attributes(ctx, path)`) and self-registers via `init()` calling `content.Register(...)` on a package-global `defaultRegistry`. `Registry.Detect(path)` tries exact-basename match first (via the optional `FilenameMatcher` interface — used by `build/dockerfile`, `repo/license`, `ignore/git`, `manifest/gomod`, etc.), then extension match, then magic-byte sniffing on the first 512 bytes. `Attributes(ctx, path)` is called *per matching file* during the walk and returns a `map[string]interface{}` of type-specific fields. Implementations check `ctx.Err()` at entry and inside any unbounded scan loop so cancellation propagates to a single in-flight file. Image types (`imagetype.go`) use `github.com/evanoberholster/imagemeta` for EXIF (camera_make/model/lens, taken_at, orientation, GPS, ISO, focal_length, f_stop, exposure_time) on JPEG/TIFF/HEIC/PNG, with a stdlib `image.DecodeConfig` fallback for width/height when EXIF is absent. Audio types (`audiotype.go`) use `github.com/dhowden/tag` for tags (artist, album, album_artist, composer, year, track, genre + reused `title`) across MP3/M4A/FLAC/OGG; the library auto-detects format internally. Playback metadata (duration, bitrate, sample_rate, channels) is hand-rolled per format in `audio_flac.go` (STREAMINFO), `audio_mp3.go` (Xing/Info + CBR fallback), `audio_ogg.go` (Vorbis ID + last-page granule), and `audio_mp4.go` (atom walker for mvhd + stsd/mp4a). No new audio libraries — the parsers are bounded (header + at most a 64 KiB tail for OGG).

Video types (`videotype.go`) cover MP4/MOV/M4V (`video_mp4.go`), MKV/WebM (`video_mkv.go`, hand-rolled EBML), and AVI (`video_avi.go`, RIFF chunks). Surfaces `duration`, `bitrate`, `video_codec`, `audio_codec`, `video_width`, `video_height`, `frame_rate`. The MP4 box-walking primitives (`mp4_box.go`: `readBoxHeader`, `walkBoxes`, `descendBoxes`) are shared with audio MP4 parsing. No CGO, no new heavy deps.

Archive types (`archivetype.go`) cover ZIP via `archive/zip` (`archive_zip.go`), TAR via `archive/tar` (`archive_tar.go`, walker exported so the tar.gz path can compose), GZIP standalone (`archive_gzip.go`, reads ISIZE footer for uncompressed size — caveat: mod 2³² for files > 4 GiB, matches `gzip -l`), and TAR.GZ (gzip-wrapped tar via `compress/gzip` → the same TAR walker). Surfaces `entry_count`, `uncompressed_size`, `top_level_entries`, `has_root_dir`. ZIP detection is magic + extension; TAR / TAR.GZ are extension-only because TAR's magic at offset 257 doesn't fit the prefix-based sniffer. The detector now does longest-suffix match (`detector.go`) so `.tar.gz` registered against `archive/tar+gzip` wins over `.gz` registered against `archive/gzip`. All stdlib — no third-party libs for archives. 7Z / RAR / BZ2 / XZ are out of scope (need third-party or non-stdlib code).

Binary types (`binarytype.go`) cover ELF via `debug/elf` (`binary_elf.go`), Mach-O thin + fat/universal via `debug/macho` (`binary_macho.go`, tries `NewFatFile` first then falls back to `NewFile`), and PE via `debug/pe` (`binary_pe.go`, type-switches between `OptionalHeader32`/`64`). Surfaces `architectures` (list — supports fat Mach-O), `bitness`, `binary_format`, `binary_type`, `is_dynamically_linked`, `is_stripped`, `entry_point`. ELF / Mach-O thin / PE are detected by magic + extension; Mach-O fat magic `0xCAFEBABE` is deliberately NOT registered (collides with Java `.class`) — fat binaries land via extension (`.dylib`) or by being parsed once detection has fired some other way. Architecture-name normalisation lives in three switch tables (one per format) returning canonical lowercase strings (`x86_64`, `arm64`, `i386`, `arm`, `ppc64`, `riscv64`, `s390x`, `mips`, `unknown`). All stdlib — no third-party libs for binaries. WASM, COFF object files, DWARF / debuglink resolution, and code-signing detection are out of scope.

Email types (`emailtype.go`) cover RFC 5322 `.eml` via `net/mail` (`email_eml.go`) and Unix mbox archives via a hand-rolled splitter (`email_mbox.go`) on top of the same `net/mail` parser for the first message. Shared header-extraction helpers live in `email_attrs.go` (`emailAttrs`, `firstAddress`, `addressList`, `stripAngles`, `decodeMIMEHeader`, `countAttachments`, `isAttachmentPart`). Surfaces reuse `title` (Subject) and `author` (From, display name preferred) for cross-family vocabulary, plus seven email-specific attributes: `email_to`, `email_cc`, `email_message_id`, `email_in_reply_to`, `sent_at`, `attachment_count`, `email_count`. Detection: `.eml` is extension-only (no canonical magic); `.mbox` is extension + 5-byte magic `From ` at offset 0. For `.mbox` archives, the per-message attributes reflect the FIRST message; `email_count` carries the archive's multi-message shape. All stdlib — no third-party libs. Outlook `.msg`, body-text extraction, encoded-attachment decoding, and DKIM/PGP verification are out of scope.

Source-code types (`sourcetype.go`) cover 19 languages with one `sourceType` struct backing all of them — per-language behaviour lives in config fields (`name`, `language`, `exts`, `lineComment`, `blockOpen`, `blockClose`). Languages: Go, Python, JS, TS, Rust, C, C++, Java, Ruby, Swift, Kotlin, Scala, Shell, Lua, Elixir, Clojure, Haskell, OCaml, Zig. Detection is extension-only — no third-party language-detection lib. The `Attributes` method scans line-by-line and classifies each as blank / comment / code based on the per-language line + block comment markers; mixed lines (code with trailing comment) count as code, lines beginning with a comment marker count as comment. Surfaces reuse `language` (programming-language name) and `line_count` (total physical lines), plus three new attributes: `loc` (non-blank, non-comment), `comment_loc`, `blank_loc`. Adding a new language is a single `registerSource(name, language, exts, lineComment, blockOpen, blockClose)` call in `init()`. Shebang detection for extensionless scripts, AST-derived attributes (cyclomatic complexity, function count), and string-aware comment classification are out of scope.

Notebook types (`notebook.go`) cover Jupyter `.ipynb` and Apache Zeppelin `.zpln` — both JSON-on-disk formats. Detection is extension-only (the longest-suffix-match detector ensures `.ipynb` / `.zpln` beat `.json` for those specific extensions). Surfaces a shared attribute set — `cell_count`, `code_cell_count`, `markdown_cell_count`, `kernel` — plus reused `language` (Jupyter `metadata.language_info.name`) and `title` (Zeppelin notebook `name`). The `notebook/*` content-type prefix lights up `is_notebook`. Per-cell language detection (Zeppelin paragraphs declare their own `%spark` / `%pyspark` interpreters), output-cell parsing, and Jupyter title heuristics (first-markdown-cell H1) are out of scope. Malformed JSON degrades silently to empty attrs — matches the broader "broken file doesn't fail the walk" pattern.

Disk-image types (`diskimagetype.go`) cover DMG (UDIF — `disk_dmg.go`, 512-byte `koly` trailer at EOF), ISO 9660 (`disk_iso.go`, Primary Volume Descriptor at offset 0x8000 with `volume_label` + `created_at`), VHD (`disk_vhd.go`, Connectix 512-byte footer at EOF with `disk_type` fixed/dynamic/differencing + `created_at`), VHDX (`disk_vhdx.go`, Hyper-V v2 with region-table-driven Metadata-region walk to extract `virtual_size` best-effort), VMDK sparse (`disk_vmdk.go`, `KDMV` header at offset 0), QCOW2 (`disk_qcow2.go`, big-endian header outlier — `cluster_bits` + `is_encrypted`), and WIM (`disk_wim.go`, `MSWIM\0\0\0` magic + `image_count`). DMG / ISO / VHD are extension-only (their signatures are at offsets the start-of-file sniffer can't reach — same pattern as TAR). Surfaces a cross-format `disk_image_format` + `virtual_size` from a shared `diskImageAttrs(...)` factory in `diskimagetype.go`, plus per-format extras packed into the same Attributes map. All stdlib — no third-party libs for disk images. Apple sparseimage / sparsebundle, reading files inside the filesystem the image contains, and OCR are out of scope.

Install-package types (`installpackagetype.go`) cover macOS XAR `.pkg` (`install_pkg.go`, `xar!` magic at offset 0 — surfaces `package_kind = "macos-installer"`), Debian `.deb` (`install_deb.go`, `!<arch>\n` ar magic — checks the first ar member is `debian-binary` to discriminate from generic ar archives), Red Hat `.rpm` (`install_rpm.go`, parses the 96-byte legacy Lead at offset 0 for `package_name` / `package_version` / `package_release` / `package_arch` / `package_kind`), and Linux `.appimage` (`install_appimage.go`, extension-only because the 4-byte AppImage marker lives at offset 8 — overlaid on the underlying ELF's e_ident padding; surfaces `appimage_version` 1 or 2). Surfaces a cross-format `package_format` from a shared `installPackageAttrs(...)` factory. All stdlib — no third-party libs. Extracting control.tar.* inside .deb to surface name+version, walking the XAR TOC, parsing the RPM Header following the Lead, and reading the SquashFS inside .appimage are out of scope for v1. `.msi` / `.snap` / `.flatpak` are also out of scope.

VM-bytecode types (`bytecodetype.go`) cover Java `.class` (`bytecode_class.go`, `0xCAFEBABE` magic — walks the constant pool to resolve `class_name` / `super_class` / `interfaces` through the `CONSTANT_Class → name_index → CONSTANT_Utf8` indirection, decodes `access_flags` into canonical strings, counts fields + methods via `skipMembers`), Python `.pyc` / `.pyo` (`bytecode_pyc.go`, extension-only because magic is version-dependent — parses the 16-byte PEP 552 header to surface `python_version` from the magic-number table and `source_mtime` for timestamp-based pycs), and WebAssembly `.wasm` (`bytecode_webasm.go` — filename intentionally avoids `_wasm.go` suffix which Go treats as a GOARCH=wasm build constraint; walks sections via LEB128-prefixed size headers, surfaces `section_count` / `import_count` / `export_count`). Surfaces a cross-format `bytecode_format` + agent-friendly `runtime_version` (e.g. "Java 17" / "Python 3.11" / "WebAssembly 1.0") from a shared `bytecodeAttrs(...)` factory. All stdlib. Defensive caps: classMaxPoolEntries=65535 (JVMS limit), classMaxStringBytes=65535, wasmMaxSections=128, wasmMaxImportsExports=1M. Fuzz targets cover all three parsers — the binary-header-walker class is high-leverage. `.jar` / `.war` / `.ear` / `.aar` still detect as `archive/zip`; their inner `.class` entries surface via `archive-contents`. `.NET` PE assemblies stay as `binary/pe` (separate code path — extending PE with CLR-metadata detection is a follow-up).

Science-data types (`sciencetype.go`) cover FITS (`science_fits.go`, `science/fits`), VOTable (`science_votable.go`, `science/votable`), HDF5 (`science_hdf5.go`, `science/hdf5`), PDS3/PDS4 (`science_pds.go`, `science/pds3` + `science/pds4`), and CDF (`science_cdf.go`, `science/cdf`) — the full set of NASA-archive-standard scientific data formats. FITS — the Flexible Image Transport System, the lingua franca of astronomy since 1981. Detection is magic (`SIMPLE  =` ASCII prefix at offset 0) plus the canonical `.fits` / `.fit` / `.fts` extensions; the 9-byte magic is the most deterministic prefix in the format and works on files with non-standard names. `parseFITSHeaders` walks the primary HDU as 80-byte ASCII keyword-value records inside 2880-byte blocks, terminated by an `END` card; `countHDUs` then advances past each HDU's data unit (size = |BITPIX|/8 × Π(NAXISᵢ), rounded up to a 2880-byte block) to count extensions in multi-extension FITS (MEF) files. Bounded by design: 256 KiB read cap, 100-HDU walker cap, NAXIS clamp at 999 (the FITS Standard §4.4.1 limit), overflow-guarded NAXISᵢ product computation. Surfaces a cross-format `science_format = "fits"` plus per-format attributes — `telescope`, `instrument`, `object`, `observer`, `date_obs`, `exptime`, `filter`, `airmass`, `ra`, `dec`, `bitpix`, `naxis*`, `hdu_count`, `fits_kind` — built via a shared `scienceAttrs(...)` factory positioned to extend over future VOTable / HDF5 / PDS / CDF additions under the `is_science_data` umbrella. Reuses cross-family vocabulary: `OBJECT → title`, `OBSERVER → author`, parsed `DATE-OBS → taken_at` (so `--group-by taken_at_year` works as it does for image EXIF timestamps). Fuzz target (`FuzzParseFITSHeaders`) covers the keyword-record walker + HDU advance with adversarial NAXIS / BITPIX seeds. All stdlib — no third-party Go deps. Pixel data is never read; only header metadata. Per-extension drilling (e.g. `hdu[1].telescope`), WCS pixel-to-sky projection, and image-data extraction are out of scope.

VOTable (`science_votable.go`, `science/votable`) — IVOA tabular standard used by every VO service (Simbad, Vizier, MAST, Gaia archive). Detection is extension-only (`.vot` / `.votable`) — the generic `<?xml` magic prefix would over-fire for plain XML, so files literally named `.xml` that contain VOTable detect as plain XML (acceptable v1 limitation per the docs). `parseVOTableHeader` is a streaming `encoding/xml` walker mirroring `body_shared.go`'s `extractXMLText` pattern: token loop with `xml.StartElement` / `xml.EndElement` / `xml.CharData` cases, namespace check against `http://www.ivoa.net/xml/VOTable` (any version), `dec.Skip()` past `<DATA>` payloads so row contents never inflate the token count. Surfaces `votable_version`, `table_count`, `total_rows` (summed from `<TABLE nrows>` attributes — row walking is out of scope), parallel `field_names` / `field_units` / `field_ucds` lists from every `<FIELD>` (capped at 10000), and `votable_data_format` from the first table's `<DATA>` child (`tabledata` / `binary` / `binary2` / `fits`). Reuses cross-family vocabulary: root `<DESCRIPTION>` → `title`, `<INFO name="creator">` → `author`. Bounded by `votableReadCap = 1 MiB` (FIELD definitions only) and the shared `maxXMLTokens = 1_000_000` guard. Fuzz target `FuzzParseVOTable` covers the XML walker with adversarial seeds.

HDF5 (`science_hdf5.go`, `science/hdf5`) — Hierarchical Data Format v5, used by LSST, LIGO, NetCDF4 (built on HDF5), every modern simulation pipeline, and PyTorch / NumPy checkpoint formats. Detection is the 8-byte PNG-style signature `\x89HDF\r\n\x1a\n` at offset 0 plus the canonical `.h5` / `.hdf5` / `.hdf` / `.he5` extensions. `parseHDF5Superblock` reads the superblock format version (byte 8) and dispatches between the v0/v1 layout (Size of Offsets at byte 13, Size of Lengths at byte 14, plus variable-length address fields) and the v2/v3 layout (compact 47-byte header with Size of Offsets / Lengths at bytes 9 / 10). Both branches defensively clamp the offset/length sizes against `hdf5MaxOffsetSize = 16`. **v1 scope is superblock-only**: surfaces `hdf5_format_version`, `hdf5_size_of_offsets`, `hdf5_size_of_lengths`. The recursive object hierarchy walk (parsing v0/v1 Symbol Table B-trees + v2/v3 fractal heaps to enumerate `group_count` / `dataset_count` / `top_level_groups`) is deliberately deferred — without real-world HDF5 fixtures the walker would be high-risk vs the metadata payoff. Bounded by `hdf5ReadCap = 64 KiB`. Fuzz target `FuzzParseHDF5Superblock` covers the version-dispatched parser with adversarial seeds. Superblocks at non-zero file offsets (the spec allows 512 / 1024 / 2048 / ... but they're rare) detect by extension only.

Database types (`databasetype.go`'s `databaseAttrs(...)` factory) cover SQLite v3 (`database_sqlite.go` + `database_sqlite_btree.go`, `database/sqlite`) — the most-deployed database in the world. Detection is magic (16-byte `SQLite format 3\0` prefix at offset 0) plus the `.db` / `.sqlite` / `.sqlite3` / `.db3` extensions. `parseSQLiteHeader` walks the 100-byte SQLite v3 header per the spec (https://www.sqlite.org/fileformat.html §1.3); surfaces `sqlite_page_size` (with the 0x0001 → 65536 sentinel transparently decoded), `sqlite_format_version` (1 legacy / 2 WAL), `sqlite_page_count`, `sqlite_schema_version` (the `PRAGMA schema_version` cookie), `sqlite_text_encoding` (`utf-8` / `utf-16le` / `utf-16be`), `sqlite_user_version` (app-managed via `PRAGMA user_version`), `sqlite_application_id` (app-stamp magic — e.g. Firefox places.sqlite uses 0x0FACADE0). Schema introspection landed as the natural follow-up (database_sqlite_btree.go): a hand-rolled SQLite b-tree walker over the `sqlite_master` table reads every CREATE statement and surfaces `sqlite_table_count` / `_view_count` / `_index_count` / `_trigger_count`, `sqlite_table_names` (sorted, capped at 100), and `sqlite_schema_fingerprint` (SHA256 of sorted `(type, name, sql)` tuples — stable across cosmetic reorders). The walker is `decodeVarint` (SQLite varint spec §1.5) + `decodeRecord` (record format §2.1, with sparse column projection) + a recursive `walkSQLiteMasterPage` handling both leaf table pages (`0x0D`) and interior table pages (`0x05`) with `sqliteMaxBtreeDepth=8` and `sqliteMaxObjects=10000` defensive caps. Overflow page chains are skipped (long CREATE statements truncate at the local payload boundary — documented; most fit inline). No `modernc.org/sqlite` dep; stays pure-stdlib. SQLCipher / encrypted databases don't expose the magic and detect as binary noise — out of scope by design. `is_database` is the umbrella family flag set via the `database/` prefix block in `setTypeFlags`, positioned to extend over future DuckDB / PostgreSQL-dump / BoltDB content types. Fuzz target `FuzzParseSQLiteHeader` covers the binary header walker with adversarial seeds.

CDF (`science_cdf.go`, `science/cdf`) — NASA Common Data Format v3+, used by heliophysics missions (ACE, Wind, Cluster, MMS, Parker Solar Probe, Solar Orbiter) and the SPDF Space Physics Data Facility archives. Detection is the deterministic 4-byte big-endian magic `0xCDF30001` at offset 0 plus the `.cdf` extension; v2.x files (magic `0x0000FFFF`) are NOT registered for magic detection because the value collides with too many other formats. `parseCDFCDR` reads the CDR (CDF Descriptor Record) header at file offset 4: surfaces `cdf_version` (Version + Release + Increment joined with dots, e.g. "3.8.0"), `cdf_encoding` (canonical short name from a switch table — "network" / "sun" / "ibmpc" / "arm-little" / ... per CDF IFD §2.1), and `cdf_majority` (row / column from CDR Flags bit 1). The GDR (Global Descriptor Record) lives at the absolute file offset stored in CDR.GDROffset; `mergeCDFGDR` reads its first 64 bytes to extract `variable_count` (NrVars + NzVars) + `attribute_count` (NumAttr). The GDR fetch has two paths: (1) GDR within the initial 64 KiB read window — common for small/medium files; (2) seek-based fallback when the underlying fs.FS implements `io.Seeker` (real `os.DirFS` files do). Bounded by `cdfReadCap = 64 KiB`. Walking the ADR linked list for ISTP-convention global attributes (`TITLE` → `title`, `PI_name` → `author`, first-record Epoch → `taken_at`) is deferred to a follow-up. Fuzz target `FuzzParseCDFHeader` covers both `parseCDFCDR` and `mergeCDFGDR` with adversarial seeds (negative variable counts, oversized GDR offsets, corrupt record types).

PDS (`science_pds.go`, `science/pds3` + `science/pds4`) — NASA Planetary Data System, two distinct variants registered via the same `pdsType` struct (mirrors the bytecode / disk-image / install-package precedent: one struct, multiple `Register(...)` calls in `init()` with different `name` / `exts` / `magic` fields). PDS3 ships as PVL labels (free-form `KEYWORD = VALUE` ASCII records) in `.lbl` files, detected by the `PDS_VERSION_ID` magic prefix at offset 0. `parsePDS3Label` strips `/* ... */` block comments first then walks lines via `bufio.Scanner` with a 1 MiB buffer cap, finding the `=` separator and looking up known keywords (`MISSION_NAME`, `SPACECRAFT_NAME` / `INSTRUMENT_HOST_NAME`, `INSTRUMENT_NAME` / `INSTRUMENT_ID`, `TARGET_NAME`, `PRODUCT_ID`, `START_TIME`). PDS4 ships as XML labels in `.lblx` files (NASA PDS4 namespace — `http://pds.nasa.gov/pds4/pds/v1`); `parsePDS4Label` uses `encoding/xml.Unmarshal` into a typed `pds4Product` struct that captures the `Identification_Area` + `Observation_Area` slice we care about, then walks `Observing_System_Component` entries to discriminate Host (→ `spacecraft_name`) from Instrument (→ `instrument_name`). The namespace check on `prod.XMLName.Space` AND a `Product_Observational` root local-name check together reject non-PDS4 XML. v1 supports `Product_Observational` only; Bundle / Collection / Document variants detect but return empty attrs. `is_pds3` / `is_pds4` are per-variant predicates; `is_pds` is the cross-version umbrella (set in `setTypeFlags` per-case rather than via a prefix block — the `science/` prefix is already taken by `is_science_data`). PDS3 lacks an explicit title field, so one is synthesised from `INSTRUMENT_NAME + TARGET_NAME` for cross-family vocabulary; PDS4 lifts `Identification_Area > title` directly. Both variants parse `START_TIME` / `start_date_time` into the shared `taken_at`. `.xml`-named PDS4 files detect as plain XML (documented limitation; rename to `.lblx`). PDS3 nested OBJECT/END_OBJECT groups are not walked — only top-level keywords are read. Bounded by `pdsReadCap = 1 MiB`. Fuzz targets `FuzzParsePDS3Label` + `FuzzParsePDS4Label` cover both parsers with adversarial seeds (unbalanced comments, gigantic lines, nesting attacks, all-0xFF noise).

License detection (`repofiles.go`'s `licenseType` Attributes) reads up to 16 KiB of LICENSE / LICENCE / COPYING / UNLICENSE body and matches distinctive boilerplate phrases via case-insensitive `strings.Contains` to surface a SPDX `license_id` — MIT, Apache-2.0, BSD-3-Clause, BSD-2-Clause, ISC, GPL-2.0 / GPL-3.0 / LGPL-2.1 / LGPL-3.0 / AGPL-3.0, MPL-2.0, BSL-1.0, CC0-1.0, Unlicense (14 covering > 95% of real-world OSS). Order in `detectLicenseID` matters where boilerplate overlaps (e.g. BSD-3 vs BSD-2 are distinguished by the "Neither the name" clause; Apache vs MIT by the more specific Apache "Version 2.0" marker firing first). Falls back to `""` when no marker fires. LICENSE.md and similar markdown-named variants currently fall through to the markdown content type and don't populate `license_id` (documented in schema.go).

Test-file detection lives in `sourcetype.go`'s `isSourceTestFile(language, path)` — per-language basename patterns surface `is_test_file: true` in the source/* Attributes map. Conventions covered: Go (`*_test.go`), Python (`test_*.py` / `*_test.py`), JS/TS (`*.test.{js,mjs,cjs,jsx,ts,tsx}` / `*.spec.{...}`), Rust (`tests/*.rs` integration tests — inline `#[cfg(test)]` modules are invisible to basename matching), C / C++ (`test_*` prefix / `_test` / `_tests` suffix on any of the family extensions), Java (`*Test.java` / `*Tests.java` / `*IT.java`), Ruby (`*_spec.rb` / `*_test.rb`), Swift / Kotlin / Scala / Shell / Elixir per their respective conventions. Languages without a strong filename convention (Lua, Haskell, OCaml, Clojure, Zig) never fire — those rely on directory layout or inline annotations.

Symlink awareness lives in `internal/celexpr/evaluator.go`'s `probeSymlink(displayPath)` — an `os.Lstat` + `os.Readlink` + `os.Stat` triple that returns a `symlinkInfo{isSymlink, target, broken}` struct for any real OS path; in-memory test filesystems return zero-value via the ENOENT-on-Lstat fast path. `BuildAttributesWith` consults the probe at function entry: when the entry is a symlink-to-dir reaching the function as a leaf (`useLstatInfo` branch), `os.Lstat` info replaces `fs.Stat` info so the symlink surfaces as a regular file-shaped record (its own size + mtime), not a confusing "file with dir metadata" entry. Broken symlinks bypass the cache (no target content to validate) and the registry pass entirely. `applySymlinkInfo` sets the `IsSymlink` / `IsBrokenSymlink` typed flags on `FileAttributes` and stashes `target_path` in `Extra`. The walker's `Options.FollowSymlinks` (default false) toggles dir-symlink descent: when true, an inline `walkSymlinkDir` closure in `internal/search/walker.go` resolves the target via `filepath.EvalSymlinks`, walks it via `fs.WalkDir(os.DirFS(target), ".", ...)`, and re-anchors each sub-entry's display path under the original symlink so search results show the user-facing location rather than the resolved target. No symlink-loop detection — per issue #128 we rely on the OS to surface ELOOP via Go's WalkDir.
- **`internal/celexpr`** — wraps `cel-go`. `New(expr)` declares a fixed schema of CEL variables (the union of common file attributes + every type-specific attribute any content type might emit) and compiles the program once. `BuildAttributesWith(ctx, fsys, fsPath, displayPath, registry, opts)` is the primary attribute-extraction entry point: runs `os.Stat`, optionally consults `opts.Index` (a `(size, mtime)`-validated cache), calls `registry.Detect` + `ContentType.Attributes` only on cache miss, asynchronously stores the result, and packs everything into `FileAttributes`. `BuildAttributes(...)` is a no-cache wrapper kept for callers that don't want indexing. `Evaluate` flattens `FileAttributes` into a CEL activation, supplying zero values for type-specific vars when the matched type didn't produce them — this is required because cel-go errors on undeclared/unbound variables. `Schema()` (in `schema.go`) returns the structured docs that drive both `--list` and the MCP `list_attributes` tool — the only place attribute documentation is hard-coded.
- **`internal/search`** — `Walk` is the orchestrator. It compiles the CEL expression once, then fans out: a `filepath.WalkDir` producer feeds paths into a buffered channel; N workers (`Workers`, default `NumCPU`) pull paths via a `select` on `ctx.Done()` / channel-receive, call `BuildAttributes` + `Evaluate`, and append matches under a single `sync.Mutex`. Directory traversal errors are swallowed (returning `nil` from the WalkDir func). Cancellation is honoured at three points: the producer's send-side select, every worker's receive-side select, and inside per-file `ContentType.Attributes` calls — so a cancelled walk terminates promptly even when one file is mid-scan. The hand-rolled binary parsers (MP4 / MKV / AVI / audio-MP4 / TAR / ZIP / Mach-O / PDF XMP) thread `ctx` through their inner `for` loops AND through the shared `walkBoxes` / `descendBoxes` / `walkEBML` primitives, so a multi-GB Xcode `.app` mid-parse surrenders to a cancelled context within one box's / entry's / EBML element's worth of work — not on file-EOF. The producer also honours `Options.Excludes` (basename globs) and `Options.RespectGitignore` (root `.gitignore`) via `exclude.go`'s `excluder`, pruning matched directories with `fs.SkipDir`. Multi-root walks (`Options.Roots`) iterate each root with its own `fs.DirFS` and per-root excluder — each root's `.gitignore` is honoured independently; one job channel + worker pool serves all roots. Workers populate `Result.Snippet` when `Options.IncludeSnippet` is set and the content type is text-based (`snippet.go`'s `isTextContentType`). `Options.IncludeBody` (with `BodyMaxBytes` cap, default 1 MiB) makes `BuildAttributesWith` read the full file body for text-shaped content types AND structured-document types and surface it via the `body` CEL variable, so filters like `body.contains("X")` / `body.matches("...")` (RE2 via CEL's standard string vocabulary) run at search time. The gate is `canExtractBody` in `internal/celexpr/body.go` — text-shaped (`isTextForBody`: markdown / text / html / csv / json / xml / source/*) reads raw bytes via `io.LimitReader`; structured-document (`isStructuredBody`: office/docx / office/xlsx / office/pptx / office/odt / epub / email/rfc822 / email/mbox / pdf) dispatches to `content.ExtractBody` in `internal/content/body.go`. Office and EPUB extractors walk the ZIP envelope, stream the relevant XML (word/document.xml for DOCX, xl/sharedStrings.xml + xl/worksheets/sheetN.xml for XLSX inline+shared strings, ppt/slides/slideN.xml for PPTX, content.xml for ODT, OPF spine + (X)HTML chapters for EPUB), strip tags / styling spans, and join with newlines. The email extractor goes deeper: `mail.ReadMessage` for header parsing, `mime.ParseMediaType` to dispatch on Content-Type, `mime/multipart` for boundary walks (multipart/alternative prefers text/plain over text/html; multipart/mixed / related concatenates all non-attachment text parts), `mime/quotedprintable` + `encoding/base64` for Content-Transfer-Encoding decoding, and the existing `isMboxSeparator` for splitting mbox archives across messages so an agent can `body.contains("invoice")` across an entire inbox in one call. Attachments are skipped via the existing `isAttachmentPart` helper, and headers (Subject / From / etc.) deliberately don't appear in the body — those already surface as separate CEL variables (`title` / `author` / `email_to`). The `pdfBody` extractor reuses `pdf.NewReader` (same constructor as `pdftype.go`'s metadata path), pre-caches fonts across pages once (mirrors `Reader.GetPlainText`'s internal pattern — avoids O(pages²) ToUnicode CMap rebuilds), then iterates pages with `ctx.Err()` checks calling `Page.GetPlainText(fonts)`; per-page errors / panics are caught by a per-page defer/recover so one bad page doesn't drop the document, and the top-level defer/recover guarantees the "malformed PDF → empty body, not a crash" contract. Image-only / scanned PDFs and encrypted PDFs both surface as empty body (agents can detect via `is_pdf && size(body) == 0`), and `(cid:N)` glyph fallbacks from incomplete ToUnicode CMaps are stripped via `pdfCIDPattern` at the end of extraction. Bodies are cached in a dedicated `bodies_v1` bbolt bucket (separate from `attrs_v1` so per-bucket caps and eviction can be tuned independently) — `lookupOrExtractBody` in `internal/celexpr/body.go` consults the cache before calling `readBody` and async-Puts the result on miss, so `body.contains("X")` queries against an unchanged tree drop from minutes-on-repeat to seconds-on-repeat. `Walk` (the buffered wrapper) applies `Options.Sort`/`Options.Order`/`Options.Limit` post-collection via `SortAndLimit` (exported so the MCP search handler can reuse it). `Options.RankExpr` (issue #168) is a CEL expression returning double / int / bool that's compiled at `WalkStream` entry via `(*celexpr.Evaluator).NewRank(expr)`, evaluated per file in the worker (after the filter passes), and stashed on `Result.Rank`. When set, `Walk` overrides `Sort` to `"rank"` and defaults `Order` to `"desc"` so the more expressive primitive wins gracefully; the `compareByKey` switch in `sort.go` learns a `"rank"` case that pulls from `Result.Rank`. Composes naturally with `--semantic-query` because `similarity` is already a declared CEL variable — agents can ask for `--rank 'similarity * 0.7 + recency_bonus'` in one composed query. Named **presets** (issue #168 sub-feature B) live in `internal/search/presets.go` — each `Preset` has a `Build() PresetOptions` func that bakes a fresh `time.Now()` into time-relative CEL expressions at invocation. The catalog is data-driven (one struct literal per preset); `TestPresets_AllCompile` verifies every preset's CEL compiles. The CLI `preset <name>` subcommand and MCP `query_preset` / `list_presets` tools both translate `PresetOptions` into `search.Options`; per-call `dir` / `limit` / `excludes` flags override the preset's defaults without forking the catalogue. Partial-result **suggestions** (issue #168 sub-feature C) live in `internal/search/suggestions.go` — `SuggestionsForSearch` / `SuggestionsForStats` inspect the walk options + collected matches + elapsed time + cancellation reason and emit a slice of agent-actionable strings (bump timeout, narrow to hot directory via longest-common-prefix, `--body` warning, missing-prunes, lax-filter). Surface: MCP `SearchOutput` / `StatsOutput` / `FindMatchesOutput` carry a `suggestions []string` field on cancellation; the CLI `printSuggestions` helper writes the same hints to stderr after the existing "interrupted" / "timed out" message. The CLI search path passes `nil` matches at the cancellation site (the streaming printer has already flushed them to stdout), so the hot-directory heuristic is MCP-only there; stats and find-matches CLI paths thread the per-file paths through. `ComputeStats` reuses `Walk` and aggregates a `Stats` histogram bucketed by `Options.GroupBy` (default `content_type`; full set in `ValidGroupBys` — string keys plus time-bucket keys `mtime_year/month/day`, `taken_at_*`, `sent_at_*`, `date_*`). Time-bucket keys format the underlying `time.Time` via `time.Format`; zero timestamps bucket as `"(no date)"`. `Stats.Groups` is the bucket list; `Stats.ContentTypes` is populated alongside only for the default group_by (back-compat with v0.20). **Fast path**: when `GroupBy` is detector-only (`content_type` / `ext` / `dir` / `mtime_*`) AND the CEL expression is trivial (empty or `"true"`), `ComputeStats` sets `Options.SkipAttributesParse=true`. The walker then skips `ContentType.Attributes(...)` for every file, falling through to `registry.Detect` + `setTypeFlags` only. The index cache is bypassed for both Lookup and Put under this flag (an entry with empty Extra would poison the cache for later calls that DO want attributes). Speed difference is dramatic on `/Applications`-shape trees: minutes → seconds. Feeds both the CLI `stats` subcommand and the MCP `stats` tool. `ReadLines(ctx, fsys, path, start, end, maxLines)` (in `readlines.go`) reads a 1-indexed inclusive line range and is shared by the CLI `lines` subcommand and the MCP `read_lines` tool — capped at 1000 lines by default, with a 64 KiB per-line cap that truncates rather than failing on huge / non-UTF-8 lines. `FindDuplicates(ctx, opts, registry)` (in `duplicates.go`) is two-pass: first walk to bucket by size, then hash only files in size-collision groups via `crypto/sha256`. Hashes cache into `index.Entry.Hash` so repeat runs on unchanged trees are free. `Options.MinSize` skips small files; the duplicates output is sorted by `wasted_bytes` desc. `FindNearDuplicates(ctx, opts, registry)` (in `near_duplicates.go`) is the fuzzy counterpart: walks with `IncludeBody=true`, computes a 64-bit Charikar SimHash via `internal/fingerprint` over the extracted body of each text-shaped / structured-document candidate, then pairwise-compares fingerprints via Hamming distance (O(N²) for now; LSH banding is a future optimisation) and groups via union-find when similarity ≥ `Options.SimilarityThreshold` (default 0.85). Fingerprints cache into `index.Entry.Fingerprint` (gob-additive) merged with any existing cached fields — repeat runs skip body extraction AND the SimHash compute. The wire shape is `search.NearDuplicates` with `Groups[].Representative` (largest file in group), `Fingerprint` (hex string), and `Members[]` sorted similarity-desc. Binary content families return zero fingerprints from `fingerprint.Compute` and are silently excluded — SimHash on opaque bytes is meaningless. `WalkArchiveEntries(ctx, archivePath, opts, registry)` (in `archive_walk.go`) lets agents search INSIDE archives (ZIP / TAR / TAR.GZ / GZIP) without extracting. The archive iteration primitive lives in `internal/content/archive_iter.go` as `IterateArchive` + `ArchiveEntry` (with `Open() (io.ReadCloser, error)`). For each entry the walker reads up to `EntryReadCap` (default 8 MiB — enough to fit typical PDF / DOCX / EPUB / email bodies) of bytes into memory, wraps them in a `content.NewSingleFileFS(name, bytes, modTime, mode)` single-file fs.FS, and calls `celexpr.BuildAttributesWith` against that — reusing every existing content-type parser and CEL plumbing without a new "Detect from bytes" API. The singleFileFS's embedded `*bytes.Reader` satisfies `io.ReaderAt`, which is exactly what `openReaderAt` and the structured-document body extractors (`pdfBody`, `ooxmlBody`, `epubBody`, `emlBody`) need to operate — so PDF / DOCX / XLSX / PPTX / ODT / EPUB / RFC 5322 / mbox body extraction inside archives all just work through the existing pipeline. Per-entry attribute extraction surfaces `content_type` + the full Extra map; CEL evaluation runs against the resulting `*FileAttributes`. The `Options.Glob` filter (basename `filepath.Match`) runs BEFORE the per-entry detection as a cheap pre-prune; `Options.Expr` is the full CEL filter. `ReadFileInArchive(ctx, archivePath, entryPath, maxBytes, registry)` (in `archive_read.go`) is the single-entry counterpart — returns the bytes + detected content_type + attributes for one named entry, or `ErrArchiveEntryNotFound` when no entry matches. Caching: `index.Entry.EntryAttributes []EntryRecord` (gob-additive) stores the per-archive entry-list keyed on the OUTER archive's (size, mtime); cache-hit calls filter cached records via `celexpr.AssembleAttributes` without opening the archive. Archives with more than `ArchiveCacheMaxEntries` (10_000) entries skip the cache (encoded payload would blow the 256 KiB bbolt soft cap). `IncludeBody=true` bypasses the cache entirely — bodies aren't cached. `FindMatches(ctx, opts, registry)` (in `findmatches.go`) is the line-level grep companion: walk with `Options.Expr` to filter candidates, then for each text-typed match (`isTextContentType`) open the file and run `Options.Pattern` (RE2) line-by-line with a 64 KiB per-line cap. `Options.ContextBefore` / `ContextAfter` attach surrounding lines per hit via a ring buffer (before) and a pending-indices slice (after — the scanner keeps reading past `Options.MaxMatchesPerFile` until every pending After window is filled). Binary content types are silently dropped — line-scanning produces noise. The scanner core (`scanReaderForMatches`) takes an `io.Reader` so tests can inject hermetic input; the production path (`scanResultForMatches`) reads via `os.Open` on the absolute path that `Walk` already resolved. `WalkStream` ignores Sort/Limit — top-K is incoherent with streaming.
- **`internal/mcpserver`** — exposes the same search via the [official MCP Go SDK](https://github.com/modelcontextprotocol/go-sdk). `New(version, idx, defaultTimeout)` builds an `*mcp.Server` with the tool set (`search` thinly wraps `search.WalkStream`; `read_attributes` calls `BuildAttributesWith` for a single path and returns the same `search.Match` shape; `read_lines` calls `search.ReadLines` to return a line range from a file; `stats` calls `search.ComputeStats` and returns a histogram + totals bucketed by `group_by` (default `content_type`); `find_duplicates` calls `search.FindDuplicates` and returns sha256-keyed groups of byte-identical files; `find_near_duplicates` calls `search.FindNearDuplicates` and returns SimHash-keyed groups of similar files (catches typo / template / regeneration edits that exact-hash misses); `list_archive_contents` calls `search.WalkArchiveEntries` and exposes per-entry CEL filtering inside ZIP/TAR/TAR.GZ/GZIP; `read_file_in_archive` calls `search.ReadFileInArchive` for single-entry byte extraction; `find_matches` calls `search.FindMatches` and returns line-level regex hits with optional before/after context; `list_attributes` returns `celexpr.Schema()` plus the registered content types; `index_stats` returns the cache's hit/miss/put/stale/error counters; `detect_project` / `find_projects` / `resolve_project_for_path` surface the project-type registry — the third is the path-anchored walk-up that answers "what project does this file belong to?" without listing the whole tree). `search` and `read_attributes` accept a `fields []string` input that projects each match to only the listed attributes (path / content_type / size stay always-on); validation against `search.MatchFieldNames()` rejects unknown names at request time, and projection runs AFTER `sort_by` so the sort key need not appear in the response. Tool handlers are methods on a per-server `handlers` struct that holds the shared `index.Index`, the `defaultTimeout`, and the `version` string stamped at `New(...)`. Every tool output struct embeds `CommonOutput` (`internal/mcpserver/common_output.go`) so the response carries a `server_version` field — handlers set `out.ServerVersion = h.version` on the success path, giving agents a self-describing signal for "is the running binary actually the version I expect?" without re-reading the MCP initialize handshake. The field is `omitempty`, so `go run` builds without `-ldflags` produce responses without it; the regression test (`server_version_test.go`) constructs a server with a known stamp and asserts every tool's response carries it — adding a new tool without populating the field will fail CI. Every tool call wraps its ctx with the default timeout (search and stats additionally accept a per-call `timeout_seconds` override); on timeout the search and stats tools return the partial result set with `cancelled=true` plus `cancellation_reason` and `elapsed_seconds`, NOT an error — agents inspect the flag instead of catching exceptions. Three transport runners: `Run(ctx, version, idx, defaultTimeout)` for stdio, `RunHTTP(ctx, version, addr, path, idx, defaultTimeout)` for the modern Streamable HTTP transport (MCP 2025-03-26), and `RunSSE(...)` for the deprecated HTTP+SSE transport (MCP 2024-11-05, kept for back-compat). The HTTP/SSE runners wrap the SDK's `StreamableHTTPHandler` / `SSEHandler` in a stdlib `http.Server` and shut down on ctx cancellation. The `mcp` CLI subcommand picks the runner via `--transport` and the default timeout via `--timeout`.

- **`internal/projecttype`** — directory-granularity counterpart to `internal/content`. Each `ProjectType` advertises a `Name` (e.g. `go`, `node`, `terraform`) plus a list of `Indicator`s — three variants: `HasFile` (exact basename), `HasGlob` (filepath.Match against basenames), or `CELExpr` (CEL expression evaluated against `files` + `subdirs` `list<string>` variables, compiled eagerly at `Register` time so bad CEL fails loudly). Indicators are OR'd; ANY match counts. `Detect(fsys, dir)` reads the dir listing once (split into files + subdirs) and returns every `ProjectType` that matches — a directory can fire multiple types (Go module with `docker-compose.yml` → both `go` and `docker-compose`). `Find(ctx, root, opts)` walks recursively, reporting each matched project root. Default: stop at the first match per branch; `opts.Nested = true` keeps descending for monorepo workspaces. `ProjectResolver` (in `resolver.go`) answers "what project does this file belong to?" by walking up from the file's parent dir; cached per-dir via `sync.Map` so the second file in the same project is a hit, not a fresh `ReadDir`. Wired into `celexpr.BuildOptions.ProjectResolver` and surfaced as `project_types` / `project_type` CEL variables when `search.Options.ResolveProjects` is set (CLI `--resolve-projects`, MCP `resolve_projects: true`). The single-shot, no-cache, unbounded counterpart `ResolveForPath(filePath, reg)` returns `(matchedDir, []Match)` for one-off lookups — backs the CLI `which-project` subcommand and the MCP `resolve_project_for_path` tool. Each `ProjectType` also carries a `BuildExcludes []string` field listing the canonical artefact basenames (`vendor` for Go, `node_modules` for Node, `target` for Rust / Maven, `__pycache__` etc. for Python). `CollectBuildExcludes(ctx, root)` does a `Find` pass + union and feeds the walker's basename excluder when `search.Options.PruneBuildArtefacts` is set — drives the CLI's `--prune-build-artefacts` / MCP's `prune_build_artefacts: true`. Custom project types loadable via YAML config (`config.go`'s `LoadFromFile`) — the CLI's `--project-type-config` global flag loads before kong dispatch so the types appear in every project-aware surface. Auto-discovery (`discovery.go`'s `LoadDiscovered`) searches two standard locations on every invocation: `os.UserConfigDir()/file-search-on/project-types.yaml` (user-wide, platform-specific: XDG / `~/Library/Application Support` / `%APPDATA%`) and `./.file-search-on/project-types.yaml` (per-project, CWD only — git-style walk-up is a follow-up). `--no-config-search` opts out for hermetic invocations. Backs the CLI `detect-project` / `find-projects` / `which-project` subcommands and the MCP `detect_project` / `find_projects` / `resolve_project_for_path` tools. 18 hardcoded built-ins (`go`, `node`, `rust`, `python`, `ruby`, `java-maven`, `java-gradle`, `dotnet`, `terraform`, `docker-compose`, plus static-site generators `hugo`, `jekyll`, `eleventy`, `astro`, `gatsby`, `mkdocs`, `docusaurus`, `pelican`) live in `builtins.go`. The `is_static_site` family CEL predicate (declared in `internal/celexpr/evaluator.go`'s `staticSiteTypes`) fires for any matched SSG type — opt-in via `--resolve-projects` / `resolve_projects: true` like `project_type` / `project_types`.
- **`internal/index`** — optional cache of per-file `(content_type, attributes)` keyed by absolute path and validated by `(size, mtime)`. Two implementations behind one `Index` interface: in-memory (`NewMemory`, sync.RWMutex + map; used by MCP auto-on) and on-disk (`Open(path)` / `OpenWith(path, BodyCacheCap)`, single-file bbolt with buckets `attrs_v1` (attributes), `bodies_v1` (extracted document bodies), `body_access_v1` (LRU access timestamps), and `meta`; used by the CLI's `--index-path` opt-in). Encoding is gob — JSON would mistype every int attribute (cel-go's adapter cares about `int64` vs `float64`). bbolt writes go through a single writer goroutine that batches via `db.Batch`; workers `Put` non-blocking and never throttle the walk. A 256 KiB soft cap on encoded `Entry` defends against pathological frontmatter. The body cache has its own 8 MiB per-`BodyEntry` cap and a configurable total-size cap (default 256 MiB, `BodyCacheCap.MaxBytes`); bodies bigger than 8 MiB bump `BodyOversize` and don't cache. FIFO eviction by access timestamp (touched on every successful `LookupBody` via a small write to `body_access_v1`) runs lazily inside the body-writer batch when the running total (mirrored to `meta.bodies_total_size` so it survives restarts) exceeds the cap. Body writes use a separate channel + writer goroutine from attribute writes so a body burst doesn't starve attribute throughput. `BodyCacheCap.Disable` makes the cache a no-op for callers that don't want body caching. Schema is versioned (`schema_version: 1` JSON in a `meta` bucket); the body buckets are schema-additive (a pre-body-cache v1 file gains them on first open by the new binary, no `ErrSchemaMismatch`). Stats counters (`Hits`, `Misses`, `Puts`, `Stales`, `Errors`, plus `BodyHits`, `BodyMisses`, `BodyPuts`, `BodyStales`, `BodyEvictions`, `BodyOversize`, `BodyErrors`) drive the CLI footer line and the MCP `index_stats` tool.

The CLI (`cmd/file-search-on/main.go`) uses `kong` for argument parsing. Two subcommands: `search` (default `withargs`) and `mcp`. `main()` builds a cancellable `context.Context` via `signal.NotifyContext(..., os.Interrupt, syscall.SIGTERM)` and threads it into both subcommands via `kong.BindTo`, so Ctrl-C / SIGTERM trigger graceful shutdown all the way down (HTTP `Shutdown`, walker workers exit, in-flight `Attributes` calls return `ctx.Err()`). After `Walk` returns, search results are sorted by path and dispatched to one of five printers in `cmd/file-search-on/output.go`: `bare` (paths only), `default` (`<path>\t[<content-type>]\t<size> bytes`, with count to stderr), `verbose` (multi-line indented records), `json` (NDJSON, one object per line), or a Go `text/template` via `--format`. The verbose / JSON / template paths set `search.Options.IncludeAttributes = true` so `Walk` keeps the full `FileAttributes` on each `Result`. The single wire shape lives in `internal/search/match.go` as `search.Match` — both the CLI's `-o json` / `--format` paths and the MCP `search` / `read_attributes` tools project `search.Result` into it via `search.MatchFrom`. Clients see the same fields whether they call via CLI JSON or MCP.

### Releases

Releases are tag-driven. Pushing a tag matching `v*` to `origin` triggers `.github/workflows/release.yml`, which runs **GoReleaser v2** (pinned `~> v2`, currently v2.15.4) and produces three artifacts in one shot:

1. **GitHub Release** with archives for `linux`/`darwin`/`windows` × `amd64`/`arm64` (six total — `.tar.gz` for unix, `.zip` for windows) plus `checksums.txt`. Binaries are stamped with `version`/`commit`/`date` via `-ldflags -X`; `./file-search-on --version` reads them back.
2. **OCI image** at `ghcr.io/richardwooding/file-search-on:<version>` (and `:latest` for non-prerelease tags), built by `ko` from the same `builds:` block. Linux only — `amd64` + `arm64` manifests. Base image is `cgr.dev/chainguard/static` (no shell). Image labels follow OCI annotation conventions.
3. **Homebrew cask** committed to [`richardwooding/homebrew-tap`](https://github.com/richardwooding/homebrew-tap) at `Casks/file-search-on.rb`. Install path is `brew install richardwooding/tap/file-search-on`.

Notes for future agents:

- The config uses **`homebrew_casks`**, not `brews`. `brews` was deprecated in GoReleaser v2.10; we deliberately picked the modern key. The cask form handles macOS-intel/macOS-arm/Linux-intel/Linux-arm splits via Homebrew's `on_macos`/`on_linux` DSL — works fine for CLI binaries even though casks were originally for GUI apps.
- ko publishes via [`go-containerregistry`](https://github.com/google/go-containerregistry); no Docker daemon is required on the runner. The workflow logs into `ghcr.io` with the workflow's auto-issued `GITHUB_TOKEN` (so `permissions: packages: write` is mandatory in the workflow).
- Pushing the cask to a *different* repo (`richardwooding/homebrew-tap`) requires a Personal Access Token with content write on that tap, exposed as the repo secret `HOMEBREW_TAP_GITHUB_TOKEN`. The default `GITHUB_TOKEN` is scoped to the source repo only.
- The first-ever release also needs the `ghcr.io/richardwooding/file-search-on` package's visibility flipped to public manually in GitHub package settings — otherwise `docker pull` requires auth.

#### Local dry-run

```sh
goreleaser check                                          # validate the YAML
goreleaser release --snapshot --clean --skip=publish      # builds dist/, no push
```

The snapshot run will *try* to load OCI images into the local Docker daemon and fail if there isn't one — that's expected on a dev machine and irrelevant for CI. Add `--skip=ko` to bypass it. `dist/` is gitignored.

#### Cutting a release

```sh
git tag v0.1.0
git push origin v0.1.0
```

Then watch the **Release** workflow in GitHub Actions. To roll back a botched release: delete the tag (`git push origin :refs/tags/vX.Y.Z`), delete the GitHub Release in the UI, untag the ghcr.io image, revert the homebrew-tap commit. Cheaper to cut `vX.Y.Z+1` if the change is non-destructive.

### Markdown front-matter (primary feature)

`internal/content/frontmatter.go` is the parser. `splitFrontmatter(data []byte) (*Frontmatter, []byte)` is the only entry point — it returns the parsed metadata (or `nil` if absent) and the body bytes that the rest of `markdown.go` should treat as the document.

Detection is purely by the leading bytes: `---\n` ⇒ YAML, `+++\n` ⇒ TOML, `{` at byte 0 ⇒ JSON. There is intentionally no magic-byte fallback — if the first bytes don't match, `frontmatter_format` is `""` and every promoted variable holds its zero value. Malformed input degrades silently (returns nil + the original bytes) so a broken front-matter block doesn't make a file vanish from results.

Seven keys are promoted to first-class CEL variables in `markdown.go`'s `Attributes`: `title`, `author`, `language`, `tags`, `categories`, `draft`, `date`. Anything else is reachable through `frontmatter.<key>`. Title precedence is *front-matter > H1*. `tags` and `categories` accept either a list or a bare string (which is wrapped) — that coercion lives in `stringListValue`. `date` accepts native `time.Time` (TOML) or several string layouts (`timeValue`); add new layouts there, not at call sites.

Map values are normalised through `normalizeMap`/`normalizeValue` so cel-go sees `map[string]any` end-to-end. yaml.v3 already does this for the top-level map, but nested maps from generic `any` paths can sometimes be `map[any]any`; the normaliser covers that.

When you change the promoted-variable set:

1. Update `markdown.go` `Attributes` to populate the new key in the returned map (with a zero-value fallback in the same default block — never leave a key undeclared).
2. Add a matching `cel.Variable(...)` in `celexpr.New` **and** a zero-value default in the activation map in `celexpr.Evaluate`. cel-go errors on undeclared/unbound variables at runtime, so the default is required even though the Extra-key overlay (a single `maps.Copy(activation, attrs.Extra)`) handles populated values automatically. The Extra key must match the CEL variable name verbatim; rename at the content-type source if they diverge.
3. Add an entry to `celexpr.Schema()` in `internal/celexpr/schema.go` (this drives both `--list` output and the MCP `list_attributes` tool — `printHelp` is now data-driven and needs no edits) and update the README front-matter table.
4. Add a test in `internal/content/frontmatter_test.go` that exercises the new promotion across at least one of the three formats.
5. **Update `examples/markdown.md`** — add at least one recipe that uses the new key in a CEL expression, alongside the existing `tags` / `draft` / `frontmatter.<key>` examples.

### Adding a new content type

For extensionless or exact-name files (`Dockerfile`, `Makefile`, `LICENSE`, `.gitignore`, `go.mod`, `package.json`, `.DS_Store`, `Thumbs.db`, `.directory`, …), also implement the optional `FilenameMatcher` interface (`Filenames() []string`). The detector's basename-exact-match pass runs **before** the extension pass, so exact-name types take precedence over extension-based ones (e.g. `package.json` detects as `manifest/node`, not generic `json`). The `setTypeFlags` function in `internal/celexpr/evaluator.go` is the single source of truth for content-type-name → `is_X` predicate dispatch; add a `case` for the new name in the per-type `switch`, AND — if the type belongs to a family (`build/*`, `repo/*`, `ignore/*`, `manifest/*`, `platform/*`, `system/<os>-*`, `system/*`, etc.) — add or rely on the matching prefix-`if` block at the bottom of `setTypeFlags`. The family block is an `if` chain rather than a `switch` so multiple prefixes can fire for one content type (e.g. `system/macos-ds-store` sets both `IsMacOSMetadata` and `IsSystemMetadata`). The `system/` family covers OS-generated metadata files: `system/macos-ds-store` / `system/macos-localized`, `system/windows-thumbs-db` / `system/windows-desktop-ini`, `system/linux-directory` — each fires its per-type predicate (`is_ds_store`, `is_localized`, `is_thumbs_db`, `is_desktop_ini`, `is_kde_directory`), an OS-specific family predicate (`is_macos_metadata` / `is_windows_metadata` / `is_linux_metadata`), and the cross-OS `is_system_metadata`.

1. Create a new file in `internal/content/` implementing the `ContentType` interface and call `Register(&yourType{})` from `init()`. For exact-name dispatch, also implement `Filenames() []string` (no interface declaration needed — the detector uses a type assertion).
2. If it introduces new attributes, declare matching `cel.Variable(...)` entries in `celexpr.New` **and** add a zero-value default to the activation map in `Evaluate` (`internal/celexpr/evaluator.go`). The Extra-key overlay is done in one shot via `maps.Copy(activation, attrs.Extra)`, so the per-attribute key in Extra must match the CEL variable name verbatim (e.g. emit `"img_width"` in Extra, not `"width"`). Add an `AttributeDoc` to the right slice in `celexpr.Schema()` so the new attribute shows up in `--list` and the MCP `list_attributes` tool.
3. If it's an image-family type, also extend the `strings.HasPrefix(contentTypeName, "image/")` branch logic in `BuildAttributes`. The registered-types listing in `--list` is generated from `content.DefaultRegistry().Types()`, so the new type appears there automatically.
4. For office documents, register the type with name `office/<format>` (e.g. `office/docx`); the `strings.HasPrefix(contentTypeName, "office/")` branch in `BuildAttributes` will set `is_office = true` automatically. DOCX/XLSX/PPTX use `docProps/core.xml` (Dublin Core); ODT uses `meta.xml`. Both go through the shared `readZipDublinCore` helper in `internal/content/dublincore.go`.
5. **Update `README.md`** — add the new attribute(s) to the matching family table under the *Available attributes* section (Common / Document / Data / Markdown front-matter / Images / Audio / Video). The README is the human-facing summary readers find first; `--list` is canonical but readers don't always run it.
6. **Update `examples/`** — add a recipe to the matching `examples/*.md` file (`audio.md`, `video.md`, `images.md`, `markdown.md`, `data.md`, `text.md`, `office.md`, `epub.md`). At least one concrete `file-search-on '<expr>'` invocation that uses the new attribute. If the new feature closed a "Known limitations" / "tracked in #X" note in any examples file, **delete the stale note** as part of the same PR — leaving it in misleads readers about what the tool can do.

### Adding a CEL function

CEL functions (callable from inside expressions, distinct from variables) are wired in `internal/celexpr/functions.go`. The pattern:

1. Implement the algorithm as an exported pure-Go function (e.g. `Levenshtein`, `Soundex`) — exported so it's directly unit-testable without going through the CEL environment.
2. Write a `ref.Val` binding (`*Binding` functions in `functions.go`) that adapts `ref.Val` arguments into Go primitives via `.Value().(string)` / `.Value().(int64)`, dispatches to the algorithm, and wraps the result via `types.Int(...)`, `types.String(...)`, `types.Double(...)`, or `types.DefaultTypeAdapter.NativeToValue(...)` for slices/maps.
3. Add a `cel.Function(name, cel.Overload(id, argTypes, returnType, cel.BinaryBinding|UnaryBinding|FunctionBinding(impl)))` entry to `fuzzyFunctions()` in the same file.
4. Add a `FunctionDoc` entry to `celexpr.Schema()` in `internal/celexpr/schema.go`. This is the only place function docs live — both the CLI `--list` output (via `printFuncs` in `cmd/file-search-on/main.go`) and the MCP `list_attributes` tool surface them automatically.
5. Optionally update the MCP `search` tool's `Description` in `internal/mcpserver/server.go` if the new function is significant enough to mention in the handshake.
6. Add unit tests for the algorithm in `internal/celexpr/functions_test.go` and a CEL-level integration test in the same file's `TestEvaluateFuzzyFunctions` (table-driven).
7. **Update `README.md`** — add the new function to the *Built-in functions* table (under *Available attributes*) with its signature, return type, and a one-line description. New functions also deserve a short prose recipe in the section above the table when they introduce a new query pattern (the existing fuzzy / phonetic / geographic prose is the model).
8. **Update `examples/`** — add at least one recipe under [`examples/fuzzy-search.md`](./examples/fuzzy-search.md) (or wherever the function fits thematically). Cross-link from the per-family file (`audio.md`, `images.md`, etc.) when the function is naturally used with attributes from that family.

The four call sites — algorithm impl, binding, env declaration, schema doc — all live in `functions.go` + `schema.go`. cel-go binds at runtime, so missing any of them surfaces as a CEL "undeclared reference" error during expression compilation, not at Go-build time.

#### Why README + examples/ are on the list

Steps 5 and 6 (above, and 7–8 in the function-extension flow) cover the human-facing surfaces. They're not auto-generated: `--list` reads `celexpr.Schema()` programmatically, but `README.md` and `examples/*.md` are hand-curated. v0.10.0 shipped 9 new attributes that all three programmatic surfaces (`--list`, `list_attributes`, `search.Match`) picked up correctly — and every one was missing from the README and examples until a follow-up doc-sync sweep (#61, #63). Bake the doc updates into the same PR as the code change so readers don't see drift between the canonical schema and the human summary.

### MCP server

`internal/mcpserver` is a thin adapter over the existing `search.Walk` and `celexpr.Schema()`. The `mcp` subcommand starts a stdio JSON-RPC server using the official Go SDK; nothing else in the binary changes when MCP mode is active.

Tool input/output structs (e.g. `SearchInput`, `SearchOutput`) live next to the handlers in `server.go`. JSON schemas are generated automatically from `json` and `jsonschema` struct tags by the SDK — don't hand-write the schema.

To add a new tool:

1. Define request/response structs with `json` + `jsonschema` tags.
2. Write a handler with signature `func(ctx, *mcp.CallToolRequest, In) (*mcp.CallToolResult, Out, error)`.
3. Call `mcp.AddTool(s, &mcp.Tool{...}, handler)` inside `New(...)`.
4. Add a test in `server_test.go` using `mcp.NewInMemoryTransports()` — that's how the existing tests drive the server in-process without a subprocess.

When changing the search surface, prefer adding inputs to the existing `search` tool over forking a new tool, so MCP clients see one entry point that mirrors the CLI.
